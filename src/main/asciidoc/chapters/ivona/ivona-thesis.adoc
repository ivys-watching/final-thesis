= 1 Topic "Database Architecture Strategies Focused on Storing Machine to Machine Data" Ivona Riszteszki


== Abstract
The rapid growth of real-time mobile applications has led to an increasing demand for secure, scalable and reliable storage solutions for high-frequency machine-to-machine data. +
This thesis investigates which database architecture management system architecture best supports the secure and scalable storage of real-time machine-to-machine geolocation data in a mobile application. Relational and non-relational approaches are evaluated with respect to specific requirements. +
To address future growth and technical measures for secure data storage, growth scenarios are simulated with a sharding strategy developed to support the predicted user base of the application. The thesis further presents a compliant backup architecture combining PostgreSQL-specific features such as WAL archiving, base backups and furthermore encrypted storage. +
The findings demonstrate that a relational PostgreSQL database architecture can efficiently support high-frequency geolocation workloads when properly indexed and combined with an appropriate scaling and backup strategy, while also considering performed benchmark tests. +
The work provides both theoretical analysis and practical implementation guidance for developers designing secure, real-time location-based mobile systems.

== Introduction

*Problem Statement:* Storing high-frequency machine-to-machine geolocation data securely and reliably in a mobile application

*Motivation:* Growth of real-time tracking system and practical implementation in mobile applications

*Research question:* Which database architecture strategy best supports secure, scalable, and reliable storage of real-time M2M geolocation data in a mobile application?

=== Scope of Project

The application that will be developed in the scope of this thesis work is HomeSafe, a mobile application that tracks geolocation data that is stored on a machine-to-machine basis. The requirements for the project are as follows:

* Consistency across all user accounts
* Scalability in case the application user base grows. Currently, the application is only planned to be marketed in Vienna, Austria, with plans for future expansion.
* Ease of use, as it is possible that other developer teams continue working on the application.
* Maximal security, as the app deals with real-time location data in emergency situations. This technical measure is also needed for keeping personal data of users as safe as possible.
* Reliability, since the application should have long-term support with as little data loss as possible, ideally none.

This thesis aims to compare different database management systems based on these criteria. I will consider both relational (SQL) and non-relational (NoSQL) databases.

== 1.1 Characteristics of Machine-to-Machine Data

=== 1.1.1 What is Machine-to-Machine Communication?

==== 1.1.1.1 Definition

Machine-to-machine (M2M) is defined as direct communication between two devices without human intervention, either through wires or wireless. <<_1, [1]>> M2M systems are especially important for real-time data provision as well as increased efficiency when handling data. <<_2, [2]>>

==== 1.1.1.2 Differences to Human-Generated Data

Human-generated data is made by humans, such as the contents of an email, a written document, images, audio files, etc. Conversely, machine-to-machine data is collected automatically through sensors, devices, or systems with no human intervention. This includes GPS tracking, a vending machine tracking its own stock, ATMs tracking the amount of banknotes, etc. M2M data is typically structured, high-volume and automated, while human-generated data is unstructured with a lower volume and slower.

==== 1.1.1.3 Implementation in Mobile Applications

M2M communication is typically implemented in mobile devices using cellular networks, GPS, and Wi-Fi.
For geolocation-based mobile applications, the most important features of M2M data would be:

* High write frequency
* Real-time read requirements
* Potential burst traffic in emergencies
* Sensitive personal data security

== 1.2 Theoretical Foundations

=== 1.2.1 What is a Database Management System?

A database management system (‘DBMS’) is a connection point between the user and the database, allowing the user to access, change and organize the database and the data if needed. <<_3, [3]>> Charles W. Bachmann created the first database management tool, IDS, which was tested and running by 1963. <<_4, [4]>> This set off an increased interest in database management as well as the development of new approached, most notably MySQL in 1995 <<_5, [5]>>, PostgreSQL in 1996 <<_6, [6]>> and NoSQL (“not only SQL”) in the 2000’s. <<_7, [7]>>

===== ACID Model

SQL databases are structured over the ACID model. ACID stands for atomicity, consistency, isolation and durability. According to <<_8, [8]>>, the definitions for the properties are as follows.

====== Atomicity

Atomicity ensures that all steps within a single transaction are fully completed. If they are not – for example, if the last step fails – all of them are reverted to their original state.

====== Consistency

Consistency ensures that data is following predefined rules and constraints in order to achieve integrity and correctness. <<_9, [9]>>

====== Isolation

Isolation ensures that all processes (operations and calls to the database) are independent from one another, even if they’re executed at the same time.

====== Durability

Durability ensures that the database keeps all data that has already been saved, even if the system experiences a failure. All changes must be permanent and must not be impacted by future system failures.

The most popular DBMS for SQL are MySQL and PostgreSQL. Though they are both relational DBMS, they contain different features and advantages to their use.

===== BASE Architecture

NoSQL databases use the BASE architecture. BASE stands for Basically Available, Soft state, Eventual Consistency.

====== Basically available

The system will be available, even in the event of a failure.

====== Soft state

The state of the data changes without application interaction, which is connected to eventual consistency.

====== Eventual consistency

The system will eventually be consistent after the application input or data change, but the consistency is not guaranteed immediately or at a transaction level. <<_18, [18]>>

== 1.3 Comparative Evaluation

=== 1.3.1 SQL Databases

==== 1.3.1.1 MySQL

MySQL is a database that is considered reliable and well performing with the open source fork MariaDB, a free community edition as well as a paid alternative. It is included in the popular LAMP stack (Linux, Apache, MySQL, Python/Perl/PHP). <<_10, [10]>>

===== Scalability

MySQL is considered highly scalable, with different strategies that can be implemented for best results. Indexing, query optimization, connection pooling and caching are just some of the best-practice options in order to increase good maintenance of the systems, make them more scalable and less demanding. <<_11, [11]>>

===== Ease of use

MySQL is considered easy to use, as it works with basic SQL. There is also an abundant ecosystem and documentation surrounding it, considering its vast area of implementation and maturity.

===== Security

The DBMS is regularly updated and is overall kept secure because of its maturity and community of developers. Its wide enterprise implementation makes it so there are frequent security patches releases, making it a stable choice.

===== Reliability

MySQL’s main feature remains its maturity – both from a data and a developer perspective, its long-term support makes it a reliable option.

==== 1.3.1.2 PostgreSQL

PostgreSQL is another popular DBMS option with a wide array of open-source database choices available. <<_12, [12]>>

===== Scalability

PostgreSQL can handle large amounts of data and grow without affecting query performance and data integrity. There are also suggested best practices in order to ensure better scalability, namely indexing, query optimization and identifying areas for improvement using the EXPLAIN ANALYZE command when querying. <<_13, [13]>>

===== Ease of use

PostgreSQL is still considered relatively easy to use for developers familiar with SQL. It comes with comprehensive documentation and user-friendly tools, making it an accessible DBMS to beginners as well as experienced users. <<_14, [14]>>

===== Security

With support for client authentication, access control, TLS and encryption at rest, as well as other strategies for failover and backup of data, PostgreSQL offers multiple security mechanisms that can protect data in a highly efficient way. <<_15, [15]>>

===== Reliability

By relying on SQL and having a mature community of developers, PostgreSQL remains a predictable and reliable option. <<_16, [16]>> Some users cite personal experience with the lack of data loss in PostgreSQL when compared to other DMBS. <<_17, [17]>>

=== 1.3.2 NoSQL Databases

==== 1.3.2.1 MongoDB

MongoDB is an open source, nonrelational DBMS. It uses documents and collections of documents as basic units of data. The format it uses is Binary JSON.<<_19, [19]>> MongoDB got released in 2009, making it the newest option for a DBMS in this thesis.

===== Scalability

MongoDB supports horizontal scaling through sharding, distributing data across multiple servers. It is able to keep up a high performance even as data volume grows. However, sharding might be a complex process with MongoDB. Improper shard key selections can result in uneven data distribution.<<_20, [20]>>

===== Ease of use

Even though MongoDB is often criticised when it comes to its application in more complex applications, the ease of use for the DBMS makes some industry experts recommend it as the standout choice for junior developers. With this DBMS; the developers have less to worry about when it comes to complexity – it uses embedding instead of joins and has simpler queries overall.<<_21, [21]>>

===== Security

MongoDB has built-in TLS/SSL encryption, role-based access control, auditing features, query history and session tracking, making it a viable option if the developer configures security features.<<_22, [22]>>

===== Reliability

While MongoDB has an active community that generally answers queries fast, several issues have been noted in past releases of the DBMS. Stale and wrong reads are noted to be a problem, as some read operations retrieve outdated information. It was first released in 2009, making it a relatively newer option.<<_23, [23]>>

==== 1.3.2.2 Redis

Redis (REmote DIctionary Server) is an open-source, in-memory NoSQL key-value DBMS. As it is an in-memory data store, it should offer extremely low latency and high throughput compared to traditional disk-based databases. It is schema-less and optimized for speed.<<_24, [24]>>

===== Scalability

Redis supports both horizontal and vertical scaling through Redis Cluster (sharding data across nodes for write/read scaling) and replicating data by reading from multiple replica nodes for scaling read queries.<<_25, [25]>>

===== Ease of Use

Redis is overall considered easy to use, especially recommended in the industry for having similar data structures as programming languages that a lot of developers would already be familiar with, such as hash tables, lists/queues and sets. It is a simple to use and implement option. <<_26, [26]>>

===== Security

Redis has built-in ACLs, authentication TLS support (encryption) and is in “protected mode” by default. Redis Enterprise Features also include audit logs, role-based access control and encryption at rest. <<_27, [27]>>

===== Reliability

As Redis is designed primarily for speed, there is a risk of data loss. Improper configuration or server crashes can result in losing data that had not yet been written to disk. As an in-memory database, Redis could consume more RAM that possibly available, possibly triggering OOM (Out of Memory) kills, crashing the instance. Scalability and memory limits are of importance when first starting with Redis. <<_28, [28]>>

=== 1.4.3 Benchmark Testing & Growth Simulation

==== 1.4.3.1 B-Trees
In order to understand and interpret the scaling of the database, a basic understanding of B-trees is relevant.
B-trees are self-balancing search tree data structures. They maintain sorted data and allow logarithmic time complexities (_O(logn)_) for search, insertion and deletion. They minimize disk I/O by storing multiple keys in a single node. They ensure that all leaf nodes are at the same depth. All of these factors contribute to B-trees significantly improving database performance, making them the standard for indexing large datasets. <<_29, [29]>>

==== 1.4.3.2 Applied Benchmark Evaluation
===== Parameters
The parameters that will be analyzed are produced by pgbench. Each parameter describes an aspect of workload configuration or system performance.

* Scaling factor - the scaling factor determines the size of the standard pgbench dataset. The smallest predefined dataset size is 1. In this project, since a custom schema and dataset were used instead of the default pgbench schema, the scaling factor does not directly influence table size. It remains at 1 because the workload logic is manually defined through the benchmark script.
* Query mode - A prepared query mode indicates that prepared statements were used. These statements are parsed and planned once, reused multiple times during execution, and overall improve performance while reflecting real-world scenarios.
* Number of clients - The number of clients indicates the number of concurrent client connections simulated during the benchmark. This simulates, in the context of HomeSafe, concurrent mobile users interacting with the system. Higher values increase contention and stress test locking, indexing, and I/O.
* Number of threads - The number of threads refers to the number of concurrent worker threads that pgbench uses to simulate the clients. This is ideally aligned with available PCU cores.
* Maximum number of tries - The maximum number of tries being one indicates that a failed transaction is not retried. If a transaction fails, it is counted as failed immediately.
* Duration - The duration indicates how long a benchmark runs. A duration of 120 seconds was chosen to allow performance stabilization and to produce statistically meaningful average results.
* Number of transactions actually processed - This indicated the total number of completed transactions during the test period. A transaction is one execution of the benchmark script logic. This metric equals the total workload throughput.
* Number of failed transactions - This shows how many transactions failed in total.
* Latency average - Latency represents the average time required to complete a single transaction, measured in milliseconds. Lower latency is preferred as it indicates better responsiveness.
* Initial connection time - This measures the time required to establish the initial database connections before transactions begin.
* TPS - Transactions per second are measured by dividing the total transactions by the benchmark duration while excluding initial connection time. TPS is the primary indicator of throughput, and should be as high as possible.

=== 1.4.3 Planned Sharding Strategy
Horizontal partitioning (sharding by user_id) would distribute the working set across multiple nodes, reducing index depth and improving cache locality, thereby restoring near-linear scalability.

==== 1.4.3.1 Shard Key Selection
The system is highly user-centric. Therefore, it makes sense that hash(user_id) would be the chosen shard key. This results in an even distribution, minimal cross-shard joins and a reduction of shard hotspots.

==== 1.4.3.2 Theoretical Shard Layout
The formula for the total theoretical throughput:
----
Total TPS ≈ TPS_per_shard × number_of_shards
----
Performance per shard should approximate ~1000 TPS.

==== 1.4.3.3 Effect on Index Depth
Sharding, overall, will reduce index size per node, B-tree height and random disk reads. This results in improving cache locality, memory-resident behaviour and reduced latency.

==== 1.4.3.4 Cross-Shard Queries
Friend queries may involve User objects on different shards. Possible solutions would be:

* Application-level fan-out
* Distributed PostgreSQL (e.g. Citus)
* Storing friendship edges on both shards

The trade-off would be increased complexity and an increased coordination cost, but the database would still be scalable.

=== 1.4.4 Backup Strategy
The backup strategy for HomeSafe is designed to ensure high availability, fast disaster recovery, and compliance with the system’s data retention policy. Since the application processes continuous write operations (location updates), the backup architecture must support minimal data loss while avoiding unnecessary storage of expired personal data.

The chosen solution is based on the native backup mechanisms of PostgreSQL, combining continuous Write-Ahead Log (WAL) archiving with daily base backups and a rolling 30-day retention window.

==== 1.4.4.1 Continuous Archiving and Point-in-Time Recovery
PostgreSQL maintains a write ahead log (WAL). This log showcases all changes made to the database's data files and exists for crash scenarios. This way, the database can be restored in the case of the crash by "replaying" the logs. According to the official PostgreSQL documentation, this approach has significant benefits:

* Any inconsistency in the backup will be corrected by log replay, making a perfectly consistent file system backup non-essential.
* Continuous backup can be achieved by continuing to backup WAL files. This lowers the load of backing up the entire database frequently.
* Replaying the WAL entry to any specific point makes it possible for the user to get a consistent snapshot of the database at the time, making this technique support point-in-time recovery.
* A warm standby system is possible - a second machine storing the WAL files will have a current or nearly-current copy of the database at any point. <<_30, [30]>>

To comply with HomeSafe’s data retention policy, WAL archives are retained for a maximum of 30 days, matching the retention period of location and session data. Older WAL segments are automatically removed to prevent reconstruction of expired personal data.

==== 1.4.4.2 Base Backup
Pg_basebackup is a utility for creating PostgreSQL backups. It is generally faster than pg_dumpall, especially when handling larger databases. Pg_basebackup is used to create backups of the entire database cluster. +
Employing a strategy of daily base backups and retaining a maximum of 30 backups at a time would ensure a secure and fast restore and disaster recovery option while complying with HomeSafe's automatic deletion policy.

==== 1.4.4.3 Encrypted Off-Site Storage
Technical measures including encryption are necessary for personal data to be stored securely.
Backup files (base backups and WAL archives) are:

* Encrypted at rest with strong symmetric encryption (e.g., AES-256).
* Transmitted over secure channels (TLS) to prevent interception.
* Stored in geographically separate infrastructure from the primary database server.
* Protected by strict access control policies and least-privilege principles.

Encryption keys are managed separately from backup storage and rotated periodically. Access to backup storage is restricted to authorized administrative roles only, and all access events are logged for auditing purposes.

==== 1.4.5.1 Approach in Project

===== Data Creation
As this project deals with M2M data, accurate and efficient data collection is vital. Data is generated through user input, mobile device sensors and system-generated logs.

===== Data Usage
The privacy policy of the project outlines the types of data used as well as third parties that might get the user's data with the specific legal bases of processing for the usage. The data that is processed when using the application is as follows:

* Account Data: Name, email address, phone number and password (required for registration).
* Emergency Contacts: Details of trusted persons the user nominates to be notified in emergencies.
* Saved Addresses: Optional addresses to enhance routing efficiency.
* User Feedback and Reports: Messages, reports of unsafe areas, or flagged incidents.
* Usage Logs: Session activity, safety feature usage (e.g., siren, panic button), and error data.
* Device Information: Device model, operating system, app version, mobile network information.
* Location Data: Real-time GPS and network-based location for safety routing, emergency alerts, and session tracking.
* App Identifiers and Similar Technologies: The App uses device identifiers, local storage, and SDK-based analytics technologies to support authentication, improve performance, and provide essential features. We do not use traditional web cookies inside the mobile application.

The third parties that the application shares data with are:

* Auth0 – for secure authentication and account management.
* Google Maps API – to provide map displays, routing, and real-time location services.
* Push Notification Providers – to deliver critical alerts, safety notifications, and system updates.
* Analytics Providers – to analyze app performance and improve user
experience.
* Emergency Services or Trusted Contacts – only when explicitly triggered by the user or in case of imminent risk to safety.

===== Data Storage
As this thesis demonstrated, the HomeSafe project implements a relational PostgreSQL database for safe storage. Due to compliance with the Regulation (EU) 2016/679 of the European Parliament and of the Council, it is stored on safe EU-based servers and encrypted.

===== Data Archiving
After 30 days, data is either archived or deleted depending on its relevance. Archived data is retained for a longer defined period before final deletion or anonymization. For example:

* Usage logs and session data are archived after 30 days
* Incident-related data may be retained longer depending on legal obligations

Only data that is no longer actively required for real-time application functionality but may still be needed for legal, auditing, or analytical purposes is archived. This includes:

* Historical usage logs
* Anonymized location data
* Safety incident reports
* System performance and error logs

===== Data Purging
The data retention periods are further defined in the privacy policy of the application. The personal data is retained only as long as necessary. Account data is retained as long as the user's account remains active. Location and session data are automatically deleted after 30 days unless needed for safety incident reporting or for legal interest.
After the retention period, data will be securely deleted or anonymized. The development team implemented a combination of technical and organizational measures to protect the data, including encryption in transit (HTTPS/TLS), encryption at rest, role-based access controls, multifactor authentication for administrative
systems and regular security reviews and vulnerability assessments.

== Conclusion
By theoretically examining multiple DBMS and evaluating them through practical benchmarking on a production level, the research of this thesis has shown that PostgreSQL is the best option for a secure, reliable, consistent and scalable mobile application operating with machine to machine real-time data. While the system reaches its limits on a larger scale of users, horizontal scaling, particularly through sharding, would be efficient options for the DBMS implementation. Indexing and efficient database modelling are also important contributing factors for maximising efficiency, while frequent backups and secure data storage using encryption ensure fault isolation and global recoverability.
