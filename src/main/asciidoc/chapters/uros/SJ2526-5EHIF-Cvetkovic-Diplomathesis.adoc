== Abstract
This diploma thesis examines privacy and data protection in safety-oriented services that include location sharing. Location data is particularly sensitive, because it can reveal routines, social relationships, and private places even without additional context. The aim of this work is to show how
privacy principles and legal requirements can be applied to location-based functionality without undermining the security goals of such services.

First, core privacy concepts are introduced, including data minimization, purpose limitation, transparency, and privacy by design and by default. On this basis, the regulatory perspective is discussed with a focus on the General Data Protection Regulation and the typical obligations that arise when processing location data, such as data subject rights and the responsibilities of controllers. In
addition, a threat landscape is developed that covers security-driven risks, for example broken access
control, injection attacks, and insecure direct object references, as well as privacy-related risks such as
tracking, profiling, secondary use, and leakage through API responses.

Finally, practical technical measures for secure back-end architectures are derived. The emphasis is on
authentication and authorization based on OAuth 2.0 and OpenID Connect, including the
Authorization Code Flow with PKCE, token validation, and scoped permissions for location access. To
make the discussion concrete, these measures are illustrated using a Spring Boot REST back-end with
delegated authentication via Auth0 and a Flutter-based client, covering protection in transit and at
rest, secure logging, retention, and the handling of security incidents. The result is a structured set of
best practices and reusable patterns for responsible location data processing in safety contexts.


== Introduction

Location sharing is often presented as a practical safety feature. It can help people coordinate, confirm that someone arrived safely, or react faster when something goes wrong. At the same time, it introduces a form of data processing that is unusually invasive because even a small number of location points can reveal routines, habits, private places, and social relationships. Once such data is collected continuously or stored over time, the privacy impact increases significantly.

This creates a tension that is central to this diploma thesis: safety-oriented applications may have legitimate reasons to process location data, but that does not reduce the responsibility to protect it. In practice, privacy and security cannot be treated as separate concerns. A system that shares location in a useful way must also ensure that data collection is limited, access is restricted, defaults are privacy-friendly, and misuse is difficult both technically and organizationally.

The purpose of this thesis is to examine how privacy and data protection requirements can be applied to safety applications with location-sharing functionality without undermining their intended benefits. The work focuses on the back-end perspective, where many of the most important decisions are enforced: authentication, authorization, API design, data storage, logging, retention, and incident readiness. Rather than proposing a new theoretical framework, the thesis aims to derive practical design patterns and implementation principles that are realistic for modern application architectures.


== Chapter 1. Fundamentals of Privacy and Data Protection

Location sharing is often introduced with a positive promise it can help people coordinate, check in with each other, or react faster in situations where time matters. At the same time, it creates a continuous stream of personal data that can reveal far more than a single point on a map - especially when those points are stored over time and correlated with other identifiers. Within the EU, the General Data Protection Regulation GDPR does not treat privacy as a purely legal wrapper around a technical system, but as a set of requirements that shape how data is collected, used, stored, secured, and deleted. The most important takeaway for an engineering-focused discussion is that lawful processing is necessary, but not sufficient the GDPR also demands data minimisation, clear purpose, strong security, and demonstrable accountability. <<ref1,[1]>>

=== 1.1 Core Privacy Principles

A practical way to ground privacy and data protection is to start with the GDPRs processing principles in Article 5. These principles include lawfulness, fairness and transparency purpose limitation data minimization accuracy storage limitation integrity and confidentiality security and accountability. <<ref1,[1]>>
These principles are deliberately technology-agnostic they describe what must be true regardless of whether data is processed in a mobile client, transmitted via APIs, or stored in databases. For location- related use cases, they become architectural questions very quickly - for example, whether precise coordinates are necessary at all, who can access them, and how long such data should exist before it is deleted. <<ref1,[1]>>

==== 1.1.1 Data Minimization and Purpose Limitation

Data minimization means processing personal data that is adequate, relevant and limited to what is necessary for a defined purpose. Purpose limitation requires that data is collected for specified, explicit purposes and not reused later in incompatible ways. <<ref1,[1]>>
In location-sharing contexts, these two principles are closely linked collecting high-precision location continuously because it might be useful later is often hard to justify, and it expands the harm if something goes wrong. A smaller dataset with a clear purpose is easier to protect, easier to explain transparently, and easier to delete when the purpose is fulfilled. <<ref1,[1]>>

==== 1.1.2 Transparency, Consent, and Lawful Basis

The GDPR requires a lawful basis for processing personal data under Article 6(1) e.g., consent, contract necessity, legitimate interests, vital interests.
Choosing the legal basis is not a formality it determines what must be communicated, what controls must be offered, and what expectations users can reasonably have. <<ref2,[2]>>
Transparency is more than a long privacy policy Article 12 requires information and communication to be concise, intelligible, easily accessible, and written in clear and plain language.
This matters in location features because people need to understand when location is processed, what level of precision is used, and how sharing can be stopped or limited. <<ref4,[4]>>
If consent is used as the lawful basis, Article 7 adds specific conditions consent must be demonstrable, and withdrawing it must be as easy as giving it.
This creates a strong incentive to design consent flows and settings in a way that is simple, reversible, and resistant to dark patterns that pressure users into sharing. <<ref3,[3]>><<ref4,[4]>>

==== 1.1.3 Privacy by Design and Default

Article 25 data protection by design and by default turns privacy from a documentation task into a design constraint. It requires appropriate technical and organisational measures to be built into processing, and it requires that default settings process only the personal data necessary for the specific purpose.
The by default part is particularly relevant for location sharing because defaults operate even when users do nothing. In other words, a privacy-friendly design is not only about adding optional controls, but about ensuring that the baseline behaviour is restrained minimal data, narrow access, limited retention, and secure handling throughout the lifecycle. <<ref5,[5]>>

=== 1.2 Location Data as a Sensitive Data Category

Location data can be unusually revealing because it is not only about where a person is, but also about what can be inferred from repeated observations. The Article 29 Working Party WP29 explicitly notes that location patterns can reveal places where a person sleeps, works, spends leisure time, and which social and professional relationships exist through co-location and routines. <<ref7,[7]>>
This sensitivity increases when location information is combined with identifiers from devices and services. Recital 30 highlights that natural persons may be associated with online identifiers provided by devices, applications, tools, and protocols and that these identifiers can be used to create profiles and identify individuals. <<ref9,[9]>>

==== 1.2.1 Types and Granularity of Location Data

Location can be derived using different technologies with different precision and data flows. WP29 discusses location derived from mobile network base stations, GPS, and WiFi positioning, and points out that the accuracy varies significantly by method and environment. <<ref7,[7]>>
For example, base station positioning can be relatively rough tens of meters in dense areas up to kilometers in rural areas, while GPS can provide considerably more precise positioning under typical conditions. WiFi-based positioning can also become very accurate, especially when large databases of access point identifiers and their approximate positions exist.

==== 1.2.2 Inherent Privacy Risks of Real-Time Location Sharing

The risk profile of real-time sharing is shaped by both immediacy and persistence. WP29 notes that location services can enable providers to obtain an intimate overview of habits and patterns, which supports profiling and far-reaching inferences about private life.
Location disclosure also has a physical-world impact misuse can enable stalking, targeted harassment, or other threats that are not limited to digital harm.
Once location data is stored, secondary use and function creep become recurring risks, because new uses often appear that were not part of the original purpose. <<ref7,[7]>>

=== 1.3 Regulatory Framework

In an EU context, GDPR compliance for location-related processing can be approached as a set of non- negotiable requirements define purposes, choose a lawful basis, communicate transparently, minimise data, enforce retention limits, secure processing, and enable rights. The regulation also expects accountability, meaning it must be possible to demonstrate that these requirements are actually implemented in practice. <<ref12,[12]>>
Because location data can create high risks, the technical side of compliance is not limited to encryption or authentication alone. It includes how defaults are chosen Article 25, how communications are made clear Article 12, and how security measures are selected and tested Article 32. <<ref4,[4]>><<ref5,[5]>><<ref6,[6]>>

==== 1.3.1 General Data Protection Regulation GDPR

Article 5 provides the guiding principles for processing, and Article 6 provides the legal bases that make processing lawful. These two parts are tightly connected a lawful basis does not remove the obligation to minimize data, limit storage, and keep processing secure.
This is especially relevant for systems that would otherwise grow data collection over time. If a feature gradually expands from occasional location checks to continuous tracking and long-term history storage, both the purpose and necessity should be re-evaluated against the Article 5 principles. <<ref12,[12]>>

==== 1.3.2 GDPR Requirements for Location-Based Services

For location-based services, three requirements show up repeatedly in practice clarity about what is processed and why transparency, restraint in what is collected and retained minimization and storage limitation, and strong protection against unauthorized access integrity and confidentiality. <<ref1,[1]>>
Article 32 frames security as a risk-based obligation and mentions measures such as pseudonymisation and encryption, ensuring ongoing confidentialityintegrityavailability, and regular testing and evaluation of security measures.
This supports a mindset where back-end security controls, monitoring, and verification are part of normal operation, not exceptional work done only after incidents. <<ref6,[6]>>

==== 1.3.3 Data Subject Rights and Obligations of Controllers

Data subject rights influence system design because fulfilling them reliably requires that data is traceable, exportable where required, and deletable where required - without weakening security.
Article 12 also requires that controllers facilitate the exercise of rights and communicate clearly about requests and outcomes. <<ref4,[4]>>
A concrete example is erasure Article 17 provides a right to obtain deletion without undue delay in several cases, including when data is no longer necessary for the purposes for which it was collected. <<ref8,[8]>>
This creates a direct link between retention policies how long location records exist and implementation details whether deletion workflows also cover derived copies such as backups, caches, or analytics datasets where applicable. <<ref18,[18]>>
At the same time, rights handling must not create new vulnerabilities. Security obligations in Article 32 remain relevant during access and deletion processes, because endpoints that return personal data or trigger deletion are high-value targets and need strong protection against impersonation and abuse. <<ref6,[6]>>

== Chapter 2. Threat Landscape for Location-Sharing Applications

Location sharing raises the stakes of security work because the data being processed is personal, time-
sensitive, and easy to abuse if it leaks or is exposed to the wrong party. <<ref7,[7]>> Unlike many other data
types, location becomes more sensitive the longer it exists: a single coordinate is often harmless on its
own, but repeated points quickly turn into a pattern that reveals routines, relationships, and private
places. <<ref7,[7]>>

This chapter describes the most relevant threats for systems that process location data via APIs. It
focuses on practical failure modes that show up repeatedly in real projects - especially broken access
control, injection and input manipulation, and integration risks - because these are the issues that
tend to turn into broad data exposure incidents. <<ref10,[10]>><<ref11,[11]>>

=== 2.1 Attack Surface and Trust Boundaries

A good threat discussion starts with the question: where does untrusted input enter, and where can
sensitive data leave? In a typical connected service, the attack surface is not limited to the “main”
endpoints; it includes authentication flows, background tasks, admin tooling, and any integration points
with third parties. <<ref11,[11]>>

Trust boundaries matter because security controls often change at boundaries. Inside a service
boundary, engineers may assume “internal calls are safe,” but that assumption breaks quickly

when multiple services, shared infrastructure, or third-party calls are involved.
<<ref10,[10]>><<ref11,[11]>>

=== 2.2 Backend-Targeted Threats

Backends are attractive targets because a single flaw can expose data at scale. Attackers do not need
to reverse engineer the user interface; they can interact directly with the API and try variations of the
same request until something behaves differently. That makes “edge cases” especially dangerous:
the one endpoint that was added late, the export route that bypasses filtering, or the debug
parameter that was never removed. <<ref11,[11]>>

==== 2.2.1 Unauthorized API Access and Endpoint Exposure

Unauthorized access is often misunderstood as “someone bypassed login.” In practice, many serious
incidents happen when the attacker is correctly authenticated but is still able to perform actions or
read objects they should not be allowed to access. OWASP describes broken access control as a top
risk and explicitly mentions insecure direct object references (IDOR) as a typical example. <<ref10,[10]>>

In API contexts, the same issue is commonly discussed as Broken Object Level Authorization (BOLA):
the endpoint may be intended for legitimate users, but the server fails to verify whether the caller is
allowed to access the specific object referenced by an ID in the path or payload. OWASP’s API1:2023
entry stresses that object-level authorization checks should be considered whenever a function
accesses a data source using an ID from the user. <<ref11,[11]>>

.Snippet 1. BOLA/IDOR example
[source,http]
----
# Victim reads their own location record (allowed)
GET /api/locations/records/10041 HTTP/1.1
Host: example.invalid
Authorization: Bearer eyJ...access_token...

# Attacker changes only the object identifier (should be denied)
GET /api/locations/records/10042 HTTP/1.1
Host: example.invalid
Authorization: Bearer eyJ...access_token...
----

==== 2.2.2 Injection Attacks and Input Manipulation

Injection remains a major vulnerability category because it targets a fundamental weakness: untrusted
input being interpreted as instructions. OWASP’s injection category covers common cases such as SQL
injection and cross-site scripting and links injection weaknesses to well-known CWEs (including SQL
injection-related entries). <<ref12,[12]>>

Even when a service uses an ORM and “never writes SQL strings,” input manipulation can still be
dangerous. Filters, sorting parameters, free-text search, and JSON fields can end up being interpreted
by downstream components (database queries, templating engines, log parsers, analytics pipelines).
The practical rule is consistent: validate input early, and then handle it safely in the layer that interprets
it (parameterized queries, safe encoding, and strict parsing). <<ref12,[12]>>

.Snippet 2. Whitelist Validation Example
[source,java]
----
double lat = req.lat();
double lon = req.lon();
if (lat < -90 || lat > 90) throw badRequest("lat out of range");
if (lon < -180 || lon > 180) throw badRequest("lon out of range");
if (req.payloadBytes() > 4096) throw payloadTooLarge();
----

==== 2.2.3 Insecure Direct Object References (IDOR)

IDOR is the practical symptom that makes broken access control visible. The system exposes a direct
reference to an internal object (like a record ID), and the server trusts that reference without confirming
the caller’s permission for that specific object. OWASP explicitly calls out “providing its unique
identifier (insecure direct object references)” as a broken access control example. <<ref10,[10]>>

In a location-sharing scenario, IDOR can be especially damaging because it is easy to automate. If one
location record can be accessed by incrementing an ID, an attacker can enumerate at scale. This is why
object-level authorization must be checked consistently and centrally, not “remembered” by each
endpoint author. <<ref11,[11]>>

=== 2.3 Third-Party Integration and Supply-Chain Risks

Location-related services often rely on external providers (maps, geocoding, routing, messaging,
analytics, identity). Each integration adds a new data flow, and every data flow creates two questions:
what data is transmitted, and what data is stored afterwards. If this is not made explicit, it is easy
to leak additional identifiers or keep more history than intended simply because the integration makes it
convenient. <<ref13,[13]>>

Supply-chain risk is broader than “a malicious library.” It includes dependency updates, transitive
dependencies, container images, build pipelines, and hosted services. NIST’s Secure Software
Development Framework (SSDF) treats secure development as a lifecycle activity and includes
practices around managing components and supplier relationships to reduce vulnerabilities and limit
the impact of exploitation. <<ref13,[13]>>

=== 2.4 Privacy-Specific Threats

Some threats are privacy threats even when the system is technically secure against outsiders. A service
can enforce authentication correctly and still be privacy-invasive if it collects too much, retains too
long, or enables tracking beyond what is necessary for the stated purpose. WP29’s geolocation opinion
highlights how location patterns can reveal sensitive details such as sleeping places, workplaces, habits,
and social relationships. <<ref7,[7]>>

<<<
==== 2.4.1 Real-Time Location Tracking and Profiling

Real-time location sharing has an immediate risk (someone can locate a person right now), but the
deeper risk comes from persistence. When real-time updates become stored history, they enable
profiling: repeated observations allow inferences about routines and relationships. WP29 explicitly
notes that geolocation services can provide an “intimate overview” of habits and patterns. <<ref7,[7]>>

==== 2.4.2 Unauthorized Data Aggregation and Secondary Use

Aggregation is where “low sensitivity” data becomes high impact. Location combined with other
identifiers makes identification easier, and correlation over time makes profiles richer. Recital 30 notes
that natural persons may be associated with online identifiers and that these identifiers can be used to
create profiles and identify individuals. <<ref9,[9]>>

Secondary use is often the quiet failure mode: data collected for one purpose gradually becomes used
for analytics, monitoring, or other goals. Even if access control remains correct, the privacy impact can
grow without any single dramatic “incident.”

==== 2.4.3 Data Leakage Through API Responses

Data leakage does not always mean a database dump. A common, avoidable pattern is returning full
objects in API responses and expecting the client to hide sensitive fields. OWASP’s API3:2023
category describes “Broken Object Property Level Authorization” and explains that it combines prior
“Excessive Data Exposure” and “Mass Assignment” into a root cause: missing or improper
authorization validation at the object property level, leading to exposure or manipulation. <<ref11,[11]>>

A practical mitigation is server-side response shaping: define explicit response models (DTOs) per
endpoint and serialize only what the endpoint truly needs to return. That both reduces privacy exposure
and limits the blast radius if one endpoint is misused. <<ref11,[11]>>


== Chapter 3. Authentication & Authorization

Security controls in location-sharing systems usually fail at the boundaries between “who are you?” and “are you allowed to do this specific thing to this specific object?”. Authentication answers the first question, while authorization must be enforced per action and per resource, otherwise the system drifts into broken access control patterns such as IDOR or object-level authorization failures. <<ref10,[10]>><<ref11,[11]>>

In this thesis, the goal is not to invent a new identity system, but to use established building blocks in a way that makes authorization decisions explicit, testable, and hard to bypass. The practical perspective matters: location endpoints are high-frequency, and small inconsistencies, like one missing ownership check or one over-broad scope, can quickly turn into bulk exposure. <<ref10,[10]>><<ref11,[11]>>

=== 3.1 IAM Basics and Least Privilege

Identity and Access Management (IAM) can be treated as a design discipline: define the identities that exist, define the actions they can perform, and then build enforcement points that are consistent across endpoints. A common mistake is to think in roles only (“user”, “admin”) and stop there, even though most real systems need object-level rules (for example, a user may view their own location history, but not another user’s; a user may share location with a chosen contact, but not with everyone). <<ref10,[10]>><<ref11,[11]>>

Least privilege is the principle that each identity should have only the minimum permissions required for the intended task. The key reason this matters for APIs is that the impact of one leaked token or one overly-permissive endpoint becomes much smaller when permissions are narrow and time-bounded. <<ref13,[13]>>

==== 3.1.1 Threat model for identities

A minimal but realistic identity model for a location-sharing service typically includes:

- User (standard account). The user can update their own location, manage sharing settings, and read their own history depending on product design.
- Friend or contact (a user who was granted access). This identity can read a specific user’s shared location, but only when sharing is enabled and within the granted constraints (time window, precision level, allowed features).
- Administrator or support role (privileged access). This should be rare and auditable; it is also one of the highest-risk identities because it bypasses normal social/consent relationships.
- Service identities (backend-to-backend). These are used for internal jobs, notifications, or processing pipelines and should never have “everything” permissions just because they are internal. <<ref13,[13]>>

The main point is that “friend access” is not a role in the classic sense. It is a relationship between two users, which means it must be checked at the object level, not only at the endpoint level. <<ref11,[11]>>

==== 3.1.2 Permission design pitfalls

Three pitfalls show up frequently:

- Over-broad roles. A single role that accidentally grants “read all locations” turns debugging convenience into a systemic privacy and security risk.
- Implicit authorization. The backend assumes the client will only request “allowed” objects; this is the same failure pattern that enables IDOR and object-level authorization issues. <<ref10,[10]>><<ref11,[11]>>
- Shared admin accounts. They destroy accountability and make incident response much harder because actions cannot be attributed cleanly. <<ref13,[13]>>

=== 3.2 OAuth 2.0 and OpenID Connect

For mobile and SPA-like clients, the standard approach is usually delegated authentication: the app redirects the user to an Identity Provider (IdP), the user authenticates there, and the app receives tokens that can be presented to the backend API. OAuth 2.0 defines the framework for delegated authorization, and OpenID Connect (OIDC) builds an identity layer on top of OAuth 2.0. <<ref14,[14]>><<ref16,[16]>>

In your context (Flutter client + Spring Boot backend), this splits responsibility cleanly:

- The Identity Provider performs primary authentication (password, MFA, account recovery).
- The backend API verifies tokens and enforces authorization rules for endpoints and objects.
- The client stores and presents tokens, but is not trusted to make authorization decisions.

==== 3.2.1 Authorization Code Flow with PKCE

For public clients (mobile apps), PKCE is a practical mechanism to reduce the risk of authorization code interception by binding the authorization request to the token exchange. In other words, even if an attacker obtains the authorization code, the code alone should not be sufficient to redeem tokens without the PKCE verifier. <<ref15,[15]>>

This is a good fit when you want a clean separation: the client uses the standard flow, the IdP issues tokens, and the backend stays focused on validation and authorization.

==== 3.2.2 Token types: ID token vs access token

In OIDC-based setups, it helps to treat token roles strictly:

- ID token: proof of authentication intended for the client, used to learn who the user is.
- Access token: proof of delegated authorization intended for the API, used to decide whether the caller can access a protected resource. <<ref16,[16]>>

Mixing these responsibilities (for example, using the ID token as an API authorization token) tends to produce fragile systems and accidental over-sharing.

==== 3.2.3 Token validation, expiry, refresh

Token validation should be implemented as a strict gate before any business logic runs. At minimum, the backend should validate signature and key material, issuer, audience, expiry, and any relevant token constraints before it accepts a request as authenticated. <<ref17,[17]>>

.Snippet 3. Spring Security validations example
[source,yaml]
----
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://YOUR_TENANT.example/ # validates iss
          audiences:
            - https://api.example.invalid # validates aud
----

Example validation sketch (illustrative only):

[source,java]
----
if(!jwt.signatureValid()) reject(401);
if(!jwt.issuerEquals(expectedIss)) reject(401);
if(!jwt.audienceContains(expectedAud)) reject(401);
if(jwt.isExpiredOrNotYetValid()) reject(401);
if(!jwt.hasScope("location.read.current")) reject(403);
----

=== 3.3 Delegated Authentication and Trust Boundaries

Delegated authentication simplifies your application, but it does not remove risk; it relocates it. Your trust boundary becomes: “I trust tokens that were issued by my IdP for my API, and I distrust everything else.” That means configuration matters a lot, and small mistakes (wrong audience, accepting tokens from another tenant, weak default rules) can create confusing failures that look like “random” authorization bugs.

Practically, it also means your logs and monitoring should distinguish between:

- Authentication failures (invalid token, expired token, wrong audience)
- Authorization failures (valid token, but not allowed to access this object)

This distinction is useful both for debugging and for incident response. <<ref13,[13]>>

=== 3.4 Authorization Model for Location Data

Authorization for location data should be designed around “what is the action” and “what is the target object.” A role like “user” can tell you what broad category the person belongs to, but it does not answer whether that user may read another user’s current location at this moment. That decision is relationship-based and often time-based. <<ref11,[11]>>

A practical model combines:

- Coarse permissions (scopes / roles) to restrict which endpoints and operations are available at all.
- Fine-grained checks (object-level authorization) to restrict which specific user’s location record can be accessed or modified. <<ref10,[10]>><<ref11,[11]>>

==== 3.4.1 Object-level authorization rules

For each location-related endpoint, define an explicit rule. Examples:

- Update my location: allowed only if subject user ID matches token subject (or a controlled “acting on behalf” mechanism exists).
- Read current location of another user: allowed only if there is an active sharing relationship and the request meets its constraints (revocation state, expiry time, precision tier).
- Read history: usually stricter than “current”, because it increases inference risk; treat it as a separate capability rather than a default extension.

These rules are exactly where broken access control issues show up when they are missing or inconsistently applied. <<ref10,[10]>><<ref11,[11]>>

==== 3.4.2 Scope design examples

Scopes should stay small and meaningful. For example:

[source,text]
----
location.read.current
location.read.history
location.write
sharing.manage
----

The point is not to create dozens of scopes, but to avoid the classic “one scope that means everything” outcome. When scopes are narrow, you can also make consent more granular, which aligns better with user expectations and makes revocation meaningful in practice.


== Chapter 4. Secure Backend Architecture Patterns

A secure back-end architecture is not only a collection of controls, but a way to make the safe path the default: consistent authorization, predictable data handling, and minimal accidental exposure. For services that process location data, architecture choices influence both security risk (who can access what) and privacy impact (how much sensitive data exists, for how long, and in what form). The GDPR framing from earlier chapters matters here because security measures and “data protection by design and by default” are not optional extras; they are part of the expected engineering baseline. <<ref5,[5]>><<ref6,[6]>><<ref1,[1]>><<ref7,[7]>>

=== 4.1 Security-First API Design

Security-first API design begins with the assumption that every request is untrusted input until validation and authorization succeed. This is one reason why API failures often show up as broken access control or unintended data exposure rather than “exotic” technical problems. For location-related payloads, strict validation is especially important because minor edge cases (out-of-range coordinates, malformed timestamps, oversized payloads) can turn into confusing behavior that is hard to monitor and easy to abuse. <<ref11,[11]>>

Authorization should be designed as an explicit rule set: “who may perform which action on which object under which conditions.” OWASP’s API Security Top 10 describes Broken Object Level Authorization (BOLA) as a common failure mode when an API accesses data by an ID supplied by the client without consistently enforcing object-level checks. A practical enforcement pattern is a fixed sequence on every endpoint: authenticate token, validate request, authorize the action (e.g., scopes), authorize the object (ownership or relationship), then execute business logic. <<ref11,[11]>>

API design should also include defensive measures against automated misuse and accidental overload. Rate limits, conservative pagination, and request-size constraints reduce the blast radius of scraping attempts and make high-frequency endpoints easier to operate under real-world traffic. Finally, response shaping (server-side DTOs) helps avoid “return everything and let the client hide it,” which OWASP associates with data exposure risks at the property level. <<ref11,[11]>>

=== 4.2 Data Protection in Transit

Data protection in transit should assume hostile networks, particularly for mobile clients. TLS for all endpoints is the baseline, and it should be treated as non-negotiable for any route that processes personal data. OWASP’s Transport Layer Protection guidance also emphasizes serving content over HTTPS and avoiding insecure transport configurations that enable downgrade or interception scenarios. <<ref6,[6]>><<ref18,[18]>>

On top of TLS, HTTP Strict Transport Security (HSTS) can help enforce HTTPS-only interaction by allowing the host to declare that user agents must communicate with it only over secure connections. This is useful because it reduces the risk that a first request is downgraded to HTTP and then intercepted. Transport security is not “set and forget”: the IETF publishes recommendations for secure TLS/DTLS use precisely because safe configurations change as attacks and ecosystem defaults evolve. <<ref19,[19]>><<ref21,[21]>>

=== 4.3 Data Protection at Rest

Data protection at rest is about limiting impact when things go wrong: database compromise, leaked backups, or accidental internal access. Article 32 frames security as risk-based and explicitly lists measures such as encryption and pseudonymisation, and it also requires regular testing and evaluation of security measures. Separately, Article 5’s principles (especially data minimisation and storage limitation) mean that “protecting stored data” is not only about encryption; it is also about reducing what is stored and how long it exists. <<ref6,[6]>><<ref1,[1]>>

For location data, this leads to a practical architectural pattern: treat “current state” and “history” as different datasets with different risk profiles and different access rules. History typically enables stronger inferences (routines, relationships, private places), so it should not automatically exist just because current location exists. Making retention enforceable (clear retention windows, deletion jobs, and explicit separation of datasets) turns storage limitation from a policy sentence into an implementable requirement. <<ref7,[7]>><<ref1,[1]>>

Secure implementation is also a lifecycle concern. NIST’s Secure Software Development Framework (SSDF) treats security as continuous work that spans requirements, design, implementation, verification, and maintenance, which fits well with the long-term nature of privacy and security obligations. <<ref13,[13]>>

=== 4.4 Logging and Observability without Data Leakage

Logging is necessary for incident investigation and abuse detection, but it can easily become a second sensitive dataset. OWASP’s Logging Cheat Sheet recommends focusing on security-relevant events (e.g., authentication and authorization failures) and warns against logging secrets and sensitive data such as passwords and tokens. For location processing, the same principle suggests that raw coordinates and raw request/response bodies should not be logged by default, because they create long-lived traces that are easy to misuse. <<ref20,[20]>>

Structured logging helps keep logs useful without storing unnecessary personal data. In practice, that means keeping operational context (timestamp, route, status code, correlation id, reason for denial) while redacting or omitting sensitive fields. Logs should also be protected operationally - restricted access, monitored queries, and defined retention periods - because confidentiality failures in logs can expose sensitive information even when the primary database is well protected. <<ref20,[20]>><<ref1,[1]>>


== Chapter 5. Privacy-Preserving Location Processing

Location data becomes sensitive less because of a single coordinate and more because repeated points form patterns that reveal routines, relationships, and private places. Privacy-preserving processing therefore starts with reducing what is collected and what is stored, not only with “locking it down” after the fact, which aligns with GDPR principles like data minimisation, purpose limitation, and storage limitation. The “by design and by default” requirement is a useful engineering lens here: the default behavior should already be restrained, even if a user never changes any settings. <<ref7,[7]>><<ref1,[1]>><<ref5,[5]>>

=== 5.1 Minimising Precision and Frequency

Data minimisation is not just “collect fewer fields”; for location it often means choosing a lower precision or a lower update frequency that still supports the safety goal. WP29 highlights that location patterns can expose sleeping places, workplaces, habits, and social relationships, which is exactly why continuous high-precision tracking should be treated as the exceptional case, not the baseline. A practical approach is to implement “precision tiers” (e.g., exact coordinates only when strictly needed, otherwise approximate areas) and to prefer event-based updates (significant movement, entering/leaving a defined area, explicit check-ins) over constant streaming. <<ref1,[1]>><<ref7,[7]>><<ref5,[5]>>

Frequency control is just as important as precision control. If the service only needs a fresh position occasionally, storing high-frequency updates creates privacy risk without creating proportional value, and it increases impact if access control ever fails. Concrete measures include: minimum time between updates, minimum distance threshold before accepting a new point, server-side throttling per user/device, and “automatic stop” conditions (e.g., sharing expires after a short time window unless actively renewed). <<ref1,[1]>><<ref7,[7]>><<ref5,[5]>>

=== 5.2 Pseudonymisation and Controlled Linkability

Pseudonymisation aims to reduce harm by limiting direct identifiers and making linkage harder unless a controlled mechanism exists. GDPR explicitly mentions pseudonymisation as a relevant security measure, but it is also clear that pseudonymised data is still personal data if it can be linked back to a person with additional information. For location processing, the practical value is that pseudonymisation reduces accidental exposure risk (e.g., if a dataset is accessed internally for debugging or analytics) and shrinks the blast radius of some classes of leaks. <<ref6,[6]>>

A useful pattern is controlled linkability: store identity and location data in a way that does not make cross-context correlation the default. Examples include separating “who” from “where” in storage (different tables/services with strict access boundaries), using relationship-specific identifiers when sharing is enabled (so two different sharing relationships cannot be trivially joined), and rotating identifiers where long-term stability is not required. Recital 30’s point about online identifiers enabling profiling is a reminder that “technical IDs” are not neutral; stable identifiers are often enough to build a profile when combined with location history. <<ref6,[6]>><<ref9,[9]>>

=== 5.3 Consent and User Control

Transparency and user control determine whether location sharing is understandable and reversible in practice, not only “lawful on paper". Article 12 requires information to be concise and clear, which suggests that a location feature should communicate, in plain language, what is shared, with whom, at what precision, and how to stop it. If consent is used as the lawful basis, Article 7 adds a strong usability requirement: withdrawal must be as easy as giving consent, which means revocation cannot be buried behind complex menus or delayed effects. <<ref4,[4]>><<ref3,[3]>>

Granularity matters because “share location” is not one decision; it is several decisions bundled together. Practical controls include: time-limited sharing (default expiry), per-contact permissions, separate toggles for current location vs. history, and a clear “pause sharing” switch that has immediate effect. On the technical side, OAuth 2.0 / OpenID Connect based setups support a capability mindset (scopes and claims), which fits well with modelling location permissions as narrow, explicit rights rather than one broad permission that implicitly allows everything. <<ref5,[5]>><<ref3,[3]>><<ref14,[14]>><<ref16,[16]>>

=== 5.4 Retention and Automated Deletion

Storage limitation is where privacy promises either become real or quietly fail over time. If location is stored indefinitely “just in case,” the dataset becomes more sensitive with every day that passes, and the harm of any future exposure increases. A retention policy should therefore be explicit and enforced automatically: current location and short-lived operational data can have short retention windows, while history - if it exists at all - should have stricter rules and a clear justification. <<ref1,[1]>><<ref7,[7]>><<ref5,[5]>>

Deletion is not only a housekeeping task; it connects directly to data subject rights such as erasure under Article 17. Implementing deletion properly usually requires handling more than the primary database record: caches, derived datasets, and backups need to be considered so that deletion is meaningful rather than cosmetic. Article 32’s requirement to regularly test and evaluate security measures supports treating retention and deletion workflows as something

that should be verified (e.g., scheduled tests that confirm data actually disappears after the retention window). <<ref8,[8]>><<ref1,[1]>><<ref6,[6]>>


== Chapter 6. Compliance and Incident Readiness

Compliance in location-based safety services is not achieved by writing a policy once; it is maintained through repeatable processes that keep privacy decisions aligned with how the system actually behaves in production. The practical goal is to make privacy and security measurable and auditable: what is processed, why it is necessary, who can access it, how long it is kept, and which safeguards reduce risk. This chapter focuses on three operational pillars that support that goal: risk thinking (DPIA-style), handling data subject requests, and being prepared for incidents. <<ref1,[1]>><<ref5,[5]>><<ref6,[6]>>

=== 6.1 DPIA Thinking and Risk Treatment

A Data Protection Impact Assessment (DPIA) can be approached as a structured way to identify risks early, decide on mitigations, and document why the remaining risk is acceptable. Even when a formal DPIA is not explicitly required, the mindset is useful because it forces concrete answers: what data is collected, what inferences are possible, what misuse scenarios exist, and which safeguards are in place. For location processing, this matters because the highest risks often come from “normal functionality” (e.g., persistent tracking, relationship inference, misuse by an authenticated user) rather than from rare edge cases. <<ref1,[1]>><<ref6,[6]>><<ref7,[7]>><<ref11,[11]>>

In practice, “when does this become high-risk?” can be evaluated using practical criteria instead of vague labels. Examples that typically increase risk are: continuous or high-frequency collection, high precision by default, long-term history retention, sharing mechanisms that reach many recipients, processing involving vulnerable groups, and integrations that enable additional correlation with other identifiers. Another strong risk multiplier is any design that makes location history easy to query, export, or enumerate, because history turns single points into patterns. <<ref1,[1]>><<ref5,[5]>><<ref9,[9]>><<ref7,[7]>><<ref11,[11]>>

A simple way to keep DPIA thinking actionable is to maintain a small “risk register” that is updated alongside architecture changes. A compact register entry can include: processing purpose, data categories (precision, frequency), threat scenario (e.g., BOLA/IDOR on history endpoint), likelihood/impact, existing controls, remaining gaps, and a clear mitigation plan with an owner and deadline. This form of documentation supports accountability because it shows not only which controls exist, but why they were chosen and how they are maintained. <<ref6,[6]>><<ref13,[13]>><<ref11,[11]>><<ref1,[1]>>

=== 6.2 Handling Data Subject Requests

Data subject requests influence system design because rights can only be fulfilled reliably if data is traceable, retrievable, and deletable without weakening security. Article 12 places emphasis on clear communication and on facilitating the exercise of rights, which is a useful engineering requirement: workflows should be understandable, predictable, and not depend on manual “hero debugging.” A common pitfall is to treat rights handling as an administrative process only, while in reality it is also an API and data management problem. <<ref4,[4]>><<ref6,[6]>>

For deletion specifically, Article 17 (right to erasure) creates a direct link between retention design and implementation detail: deletion must be meaningful, not cosmetic. This typically requires thinking beyond the primary record: caches, derived datasets, and backups may affect how quickly deletion is reflected and what “complete deletion” can realistically mean in a system with operational constraints. At the same time, rights endpoints are high-value targets, because they can reveal data or trigger destructive actions, so they must be protected at least as strongly as any other sensitive functionality. <<ref8,[8]>><<ref6,[6]>><<ref11,[11]>>

A robust request-handling workflow can be expressed in a few concrete steps:

- Strong requester verification (avoid impersonation; treat account recovery signals and re-authentication as part of the workflow). <<ref6,[6]>>
- Clear request scoping (what data, which time range, which format, and what is excluded for security reasons). <<ref4,[4]>>
- Controlled export (consistent response models; avoid excessive data exposure and “dump everything” endpoints). <<ref11,[11]>>
- Secure deletion execution (idempotent deletion jobs, audit trail, and verification that the deletion propagated where expected). <<ref6,[6]>><<ref8,[8]>>
- Safe communication (status updates that do not leak sensitive information to unauthenticated parties). <<ref4,[4]>>

=== 6.3 Incident Response and Breach Considerations

Even well-designed systems should assume that something will eventually go wrong: a misconfiguration, a leaked credential, a vulnerable dependency, or an authorization mistake that slips through. Article 32’s framing supports this assumption by requiring not only protection measures, but also ongoing evaluation and testing, which naturally leads to incident readiness practices such as monitoring, drills, and clear operational runbooks. Incident readiness is therefore part of compliance: it reduces harm, improves detection time, and makes outcomes more predictable under pressure. <<ref6,[6]>><<ref13,[13]>>

From an engineering perspective, it helps to separate two goals that can conflict if handled carelessly: collecting enough evidence to investigate incidents, while not creating a permanent secondary dataset of sensitive location data through logging and monitoring. Operational visibility should focus on security signals (authentication/authorization failures, unusual access patterns, rate limit triggers) and on the minimal metadata needed to correlate events, not on storing raw location trails. <<ref1,[1]>><<ref20,[20]>><<ref11,[11]>>

==== 6.3.1 What you log/keep to investigate incidents without storing extra location data

Logging strategy should be designed with the assumption that logs are sensitive and attractive to attackers. OWASP’s Logging guidance explicitly warns against logging secrets such as passwords and tokens, and encourages logging security-relevant events that help investigations. For location processing, the practical implication is to avoid logging raw coordinates and raw request/response bodies by default, because they can recreate sensitive movement patterns long after the original purpose has expired. <<ref20,[20]>><<ref1,[1]>>

A good compromise is to log “what happened” rather than “what was processed.” Examples of useful, low-leakage log elements are: timestamp, endpoint name, authenticated subject identifier (internal), authorization outcome (allowed/denied), reason code (e.g., missing scope, object ownership mismatch), and a correlation id for tracing across services. If location context is operationally required (e.g., diagnosing geo-fencing bugs), it can be captured in a reduced form (coarse region or bucket) with strict retention limits and access controls, rather than storing precise coordinates. <<ref20,[20]>><<ref11,[11]>><<ref1,[1]>><<ref6,[6]>>

Retention for logs should be explicit and justified, because keeping security logs forever increases privacy exposure and operational risk without necessarily improving security outcomes. This ties back to storage limitation as a principle: operational datasets should exist only as long as they are needed for their operational purpose. <<ref1,[1]>><<ref20,[20]>>

==== 6.3.2 “Day 1” response checklist

When an incident is suspected, the first day is mostly about limiting damage and preserving enough evidence to understand what happened. A concise checklist helps reduce cognitive load and prevents teams from improvising under stress. The following actions are broadly applicable to incidents involving unauthorized access or unexpected exposure patterns in location-related services. <<ref6,[6]>><<ref20,[20]>><<ref11,[11]>>

- Triage and scope: confirm signal, identify affected endpoints/data categories, and establish a single incident owner. <<ref20,[20]>>
- Containment: block abusive sources, tighten rate limits, temporarily disable high-risk features if needed, and enforce stricter authorization checks if a bypass is suspected. <<ref11,[11]>>
- Credential and key hygiene: rotate secrets and keys that may be exposed, revoke suspicious sessions/tokens where possible, and verify issuer/audience validation behavior for tokens. <<ref17,[17]>><<ref6,[6]>>
- Evidence preservation: snapshot relevant logs and configuration states with controlled access, and avoid “clean-up” actions that destroy forensic detail. <<ref20,[20]>>
- Remediation: patch the root cause (often authorization logic or response shaping in API incidents) and add regression tests that reproduce the failure. <<ref11,[11]>><<ref6,[6]>>
- Communications: coordinate internal reporting early, and involve responsible roles for decisions about external communication and any required notifications. <<ref6,[6]>>
- Follow-up: write a brief post-incident review and feed actions back into the secure development lifecycle (coding guidelines, reviews, monitoring rules). <<ref13,[13]>>

Finally, readiness improves significantly if these steps are practiced before they are needed. Short tabletop exercises and periodic reviews of logging, retention, and access control rules often find gaps that do not appear in normal development workflows. <<ref6,[6]>><<ref20,[20]>>

== Conclusion

This diploma thesis shows that privacy in location-sharing features does not have to conflict with safety goals, as long as privacy is treated as an engineering requirement rather than a documentation afterthought. Location data is especially sensitive because even a small number of points - and even more so a stored history - can reveal routines, relationships, and private places. The key takeaway is that lawful processing is necessary but not sufficient: strong results come from data minimization, clear purpose limitation, privacy-friendly defaults, and demonstrable accountability implemented throughout the lifecycle.

On the technical side, the most relevant risks are not “exotic hacks,” but recurring API failure modes such as broken access control (BOLA/IDOR), injection and input manipulation, and excessive data exposure through overly broad responses. The countermeasures derived in this work focus on solid authentication and authorization (OAuth 2.0 / OpenID Connect, Authorization Code Flow with PKCE, strict token validation, meaningful scopes, and consistent object-level checks) combined with backend patterns that make the safe path the default (validate → authorize → execute, DTO-based response shaping, rate limits, secure logging, and enforceable retention/deletion). Using a Spring Boot REST backend with delegated authentication (Auth0) and a Flutter client makes these measures concrete and transferable to similar safety-oriented location use cases.
